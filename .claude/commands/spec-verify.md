# /spec-verify — Verify Implementation Against Spec

Replay the model's witness traces against real code (conformance), run the area's test suite, validate the traceability table against actual code/test locations, detect drift, and append the result to `verification_log[]`. Runs anytime, anywhere — no branch required, CI-friendly.

A requirement is **verified** iff its witness trace replays green against the implementation (rationale: METHODOLOGY.md → "Conformance"). Hand-written tests are additive, not load-bearing.

## Usage
```
/spec-verify [target]
/spec-verify [target] --code <path>     # override code root resolved from project.json
/spec-verify [target] --tests <path>    # override tests root
/spec-verify [target] --dimension <d>   # one of: conformance, completeness, correctness, coherence, regression
/spec-verify [target] --no-conformance  # skip trace replay (e.g. adapter not generated yet)
```

`[target]` is optional: without it, every **code-bearing** target of the active change (`last_change` in `.spec/local.json` → `.spec/changes/<slug>.json`) is verified; contract targets in the change are skipped with a note (their verification is `/spec-check`). Explicit `[target]` verifies just that one.

Refuses on explicit contract targets — contracts have no code; their verification is `/spec-check`.

## Instructions

You are the **Verifier**. You compare the spec (what should be true) against the code (what is). You don't change the code or the spec; you report.

### Step 1 — Resolve target and locations

Resolve the target set:

- Explicit `[target]` → verify just it. Doesn't alter the active change.
- No target → read `last_change` from `.spec/local.json`, load `.spec/changes/<slug>.json`. Target set = every `targets[]` entry whose area has code; skip contracts with a one-line note. Print: `Change: <slug> — verifying <n> code targets`.
- No active change → if there's exactly one area with code, use it; otherwise ask.

After each target: write `verified: true/false` into the manifest entry, in addition to the area's `verification_log[]`. Run all targets even on early failure; report together.

Read:
- `specs/<target>.json`
- `.spec/project.json` (for the area's `code_repo`, `code_path`, `tests_path`, `test_command`)
- `.spec/local.json` (for the per-dev `repo_paths[code_repo]`)

Refuse if `kind == "contract"`:
```
/spec-verify does not apply to contract targets. A contract's verification is /spec-check.
Run /spec-check <target>.
```

Resolve **code root** and **test root**:
- single-repo (no `code_repo` in project.json, or no `repo_paths` entry): paths are relative to project root.
- multi-repo: `<repo_paths[code_repo]>/<code_path>/`.

If `--code` / `--tests` flags given, override.

### Step 1a — Conformance: replay witness traces

Skip if `--no-conformance` was passed, or if the area has no `conformance` section yet (then report: "Conformance not set up. Run /spec-apply to generate the adapter + harness." and continue with the other dimensions).

Preflight (deterministic):

```bash
tools/itf_tools.py status <target>        # every REQ witnessed? traces present, valid, and FRESH?
```

`status` exits 1 on missing/invalid traces **and on STALE ones** (the model's `.qnt`/`.probes.qnt` changed since the trace was found — `witness.model_sha` no longer matches). Refuse to replay stale traces: a trace from an old model proves nothing about the current spec. Report and suggest `/spec-check <target>` first.

Then run the replay harness from the code repo root:

```bash
cd <code_root_repo> && <conformance.command>
```

The harness (generated by `/spec-apply`) loads each ITF trace from `specs/<target>/traces/` (resolve `conformance.traces_dir`), and for each trace:
1. Resets the system under test to a clean state.
2. For each step, calls the adapter method mapped to the step's action (`lastAction` ghost var) with the parameters from the param ghosts (`lastUid`, `lastSid`, …).
3. After each step, asserts the observable code state matches the trace's model state under the adapter's abstraction mapping (ghost vars excluded), and asserts every invariant predicate the adapter exposes.

**Self-test first:** confirm the tampered-trace test (`_selftest.tampered.itf.json`) FAILS as expected. If it replays green, the adapter is broken — every other result this run is void; report that as the only finding and stop the conformance dimension.

Interpret results per trace:
- **Trace replays green** → the requirement it witnesses is implemented as modeled. Mark its `requirements[].status: "verified"` and `traceability[]` entry `verified: true`.
- **Trace fails at step N** → the code diverges from the model exactly there. Report the step, the expected model state, the observed code state. This is either a code bug (fix code) or model/code abstraction mismatch (fix adapter or spec — if spec, that's drift; see Step 4).

Record counts for the verification log: `traces_replayed`, `passed`, `failed`, `failed_ids`.

### Step 2 — Three dimensions

Run completeness, correctness, coherence checks against the resolved code/test roots (unless `--dimension` restricts).

**Completeness** — for each `traceability[]` entry:
- Does `code` (e.g., `authService.ts:login`) refer to a real file:symbol in the code root?
- Does each `tests[]` entry refer to a real test:name in the test root?
- If the area declares `ui_components[]`: do they map to real component files (resolve via `architecture.module_layout`)?
- If components are declared: does each component's `implements[]` action have a code mapping?

**Correctness** — for each invariant/requirement with `quint_ref` or `quint_name`:
- The Quint guard expects certain preconditions. Does the code's corresponding function check them? (Best-effort static read; report what looks suspicious.)
- For constraints: is the named constant defined with the same value in code?

**Coherence** — broader checks:
- Are there code files in the resolved `module_layout` paths that have no `traceability[]` entry? (Extras.)
- Are there `decisions[]` entries with `kind: "architecture"` that contradict the current `architecture` section? (Stale ADRs.)
- Does the `architecture.components[].implements[]` set partition cleanly across actions?

Each finding gets a severity: PASS, WARN, FAIL.

### Step 3 — Regression (run the test suite)

Run the area's `test_command` from `.spec/project.json` against the resolved code root:

```bash
cd <code_root_repo> && <test_command>
```

Capture the result. Failures recorded per-test:
- Map each failure back to its `traceability[]` entry if possible.
- Note whether the failing test was already failing before (compare against the previous `verification_log` entry if present).

### Step 4 — Drift detection

Drift = spec-traced code modified outside `/spec-apply`, in a way that makes verification fail.

Heuristic:
1. Read git log for the area's code paths since the last `verification_log` entry's `code_sha`.
2. If there are commits that touch traced files (paths in `traceability[]`) and the current verify is failing on those tests, set `drift_detected: true`.
3. If the current verify is passing, drift didn't break anything — leave `drift_detected: false` even if there were commits.

### Step 5 — Append to verification_log

```json
{
  "date": "<ISO datetime>",
  "spec_sha": "<git HEAD of spec repo>",
  "code_sha": "<git HEAD of code repo>",
  "code_repo": "<name from project.json or null>",
  "status": "pass | warn | fail",
  "drift_detected": <bool>,
  "summary": "<one line: N traces replayed, N tests passed/failed; M coverage gaps; drift if any>",
  "findings_count": <total fail+warn>,
  "conformance": { "traces_replayed": 4, "passed": 4, "failed": 0, "failed_ids": [] }
}
```

Keep the log bounded — trim entries older than 50 runs, or summarize by month if it gets unwieldy.

Also update each `traceability[]` entry's `verified` flag.

### Step 6 — Report

```
## /spec-verify auth — 2026-05-14

Code root:  /Users/alice/work/service-api/src/auth/
Test root:  /Users/alice/work/service-api/tests/auth/
Test cmd:   pnpm test auth

Spec @ abc1234   ⇄   Code @ def5678

Conformance:     ✓ 4/4 witness traces replayed green
                   REQ-001 ✓ (4 states)  REQ-002 ✓ (3 states)  REQ-003 ✓ (6 states)  REQ-004 ✓ (4 states)
Completeness:    ✓ 6 PASS, 0 WARN, 0 FAIL
Correctness:     ✓ 4 PASS, 1 WARN (INV-001 guard could be tightened — see below), 0 FAIL
Coherence:       ✓ 3 PASS, 0 WARN, 0 FAIL
Regression:      ✓ 42 passed, 0 failed

Overall:         PASS
Drift detected:  no
```

A conformance failure renders like:

```
Conformance:     ✗ 3/4 green; REQ-003 FAILED at step 5
                   expected accountStatus[bob] = Locked after 5th login_failed
                   observed code state: bob still Unlocked (lockout fires at attempt 6 — '>' vs '>=' in authService.ts:78?)
```

If drift:

```
Overall:         FAIL
Drift detected:  YES

Findings cluster on existing requirements rather than recent additions, and git log
shows commits to traced files (authService.ts) since the last verify.

Two paths:
  1. Code is wrong (regression). Revert the offending commits.
  2. Code is right; spec is stale. Run /spec auth — it'll detect drift and walk
     you through codifying it.
```

### Step 7 — Commit

```bash
git add specs/<target>.json
git commit -m "spec(<target>): verify — <pass|fail>, <summary>"
```

Show the commit message; wait for confirmation.

### CI usage

`/spec-verify` is designed to run in CI. Per-PR pipeline:
1. Check out the PR branch.
2. Run `/spec-verify <area>` for each area touched by the PR.
3. Block merge if any returns FAIL or drift_detected.

For monorepo coverage, iterate over `areas[]` from `.spec/project.json`.

**Conformance belongs in the CODE repo's own CI too.** The replay harness is just a test file — wire `conformance.command` into the code repo's standard test run (it already runs on every code PR). Then a code change that breaks model conformance fails immediately in the code repo, without any spec tooling installed; `/spec-verify` remains the cross-checking layer that also validates traceability and drift.
