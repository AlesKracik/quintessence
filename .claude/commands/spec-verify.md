# /spec-verify — Verify Implementation Against Spec

Replay the model's witness traces against real code (conformance), run the area's test suite, validate the traceability table against actual code/test locations, detect drift, and append the result to `verification_log[]`. Runs anytime, anywhere — no branch required, CI-friendly.

A requirement is **verified** iff its witness trace replays green against the implementation (rationale: METHODOLOGY.md → "Conformance"). Hand-written tests are additive, not load-bearing.

## Usage
```
/spec-verify [target]
/spec-verify [target] --code <path>     # override code root resolved from project.json
/spec-verify [target] --tests <path>    # override tests root
/spec-verify [target] --dimension <d>   # one of: conformance, completeness, correctness, coherence, regression
/spec-verify [target] --no-conformance  # skip trace replay (e.g. adapter not generated yet)
```

`[target]` is optional: without it, every **code-bearing** target of the active change (`last_change` in `.spec/local.json` → `specs/changes/<slug>.change.json`) is verified; contract targets in the change are skipped with a note (their verification is `/spec-check`). Explicit `[target]` verifies just that one.

Refuses on explicit contract targets — contracts have no code; their verification is `/spec-check`.

## Instructions

You are the **Verifier**. You compare the spec (what should be true) against the code (what is). You don't change the code or the spec; you report.

Division of labor, same as `/spec-check`: **`tools/spec-record.py verify <target>`** runs the witness preflight, the conformance replay, the test suite, the mechanical drift check, and writes the `verification_log` entry, the `requirements[].status: "verified"` flips, and `traceability[].verified` — all from exit codes and git facts, never by judgment. **You** run the static judgment dimensions (completeness / correctness / coherence below), interpret failures, and write the report.

### Step 1 — Resolve target and locations

Resolve the target set:

- Explicit `[target]` → verify just it. Doesn't alter the active change.
- No target → read `last_change` from `.spec/local.json`, load `specs/changes/<slug>.change.json`. Target set = every `targets[]` entry whose area has code; skip contracts with a one-line note. Print: `Change: <slug> — verifying <n> code targets`.
- No active change → if there's exactly one area with code, use it; otherwise ask.

Results go into the area's `verification_log[]` only — the manifest stores no phase flags ("verified" is derived from the latest log entry vs current spec/code shas). Run all targets even on early failure; report together.

Read:
- `specs/<target>.*.json`
- `.spec/project.json` (for the area's `code_repo`, `code_path`, `tests_path`, `test_command`)
- `.spec/local.json` (for the per-dev `repo_paths[code_repo]`)

Refuse if `kind == "contract"`:
```
/spec-verify does not apply to contract targets. A contract's verification is /spec-check.
Run /spec-check <target>.
```

Resolve **code root** and **test root**:
- single-repo (no `code_repo` in project.json, or no `repo_paths` entry): paths are relative to project root.
- multi-repo: `<repo_paths[code_repo]>/<code_path>/`.

If `--code` / `--tests` flags given, override.

### Step 1a — Run the recorder (mechanical work)

```bash
tools/spec-record.py verify <target> [--code-root PATH] [--skip-conformance] [--skip-tests]
```

What it does, deterministically — you never repeat or hand-record any of it:

1. **Witness preflight** (`itf_tools.witness_status`): refuses the conformance replay while any obligation is undischarged — stale, unstamped, missing, invalid, not-run, or unjustified-skip. A trace from an old model proves nothing about the current spec; suggest `/spec-check <target>` first.
2. **Conformance replay**: runs `conformance.command` from the code repo root. The harness (generated by `/spec-apply`) replays each ITF trace step-by-step through the adapter — action from the `_lastAction` ghost, parameters from the param ghosts, state asserted after every step — and includes the tampered self-test (`_selftest.tampered.itf.json`) that must FAIL; a green tampered trace means the adapter is broken and voids the run.
3. **Test suite**: runs the area's `test_command`.
4. **Mechanical drift**: failing run + traced files (from `traceability[].code`) changed since the last entry's `code_sha` → `drift_detected: true`.
5. **Ledger**: appends the `verification_log` entry (shas via `git rev-parse`), flips `requirements[].status: "verified"` and `traceability[].verified` for witnessed REQs **only when the replay was green**, trims the log to the newest 50.

Your part of interpreting a replay failure: the harness reports the first diverging step — explain whether it's a code bug (fix code) or a model/code abstraction mismatch (fix adapter or spec; spec → that's drift, Step 4 report below).

If the area has no `conformance` section yet, the runner notes it and moves on — report "Conformance not set up. Run /spec-apply to generate the adapter + harness."

### Step 2 — Three dimensions (judgment work — yours)

Run completeness, correctness, coherence checks against the resolved code/test roots (unless `--dimension` restricts).

**Completeness** — for each `traceability[]` entry:
- Does `code` (e.g., `authService.ts:login`) refer to a real file:symbol in the code root?
- Does each `tests[]` entry refer to a real test:name in the test root?
- If the area declares `ui_components[]`: do they map to real component files (resolve via `architecture.module_layout`)?
- If components are declared: does each component's `implements[]` action have a code mapping?

**Correctness** — for each invariant/requirement with `quint_ref` or `quint_name`:
- The Quint guard expects certain preconditions. Does the code's corresponding function check them? (Best-effort static read; report what looks suspicious.)
- For constraints: is the named constant defined with the same value in code?

**Coherence** — broader checks:
- Are there code files in the resolved `module_layout` paths that have no `traceability[]` entry? (Extras.)
- Are there `decisions[]` entries with `kind: "architecture"` that contradict the current `architecture` section? (Stale ADRs.)
- Does the `architecture.components[].implements[]` set partition cleanly across actions?

Each finding gets a severity: PASS, WARN, FAIL.

### Step 3 — Interpret failures (judgment work — yours)

The runner already recorded the facts (Step 1a). What remains:

- Map test failures back to `traceability[]` entries where possible; note whether a failing test was already failing in the previous `verification_log` entry.
- On `drift_detected: true`, write the two-paths explanation (revert vs codify — see the drift report shape below).
- Findings from the three dimensions (Step 2) go in the report; they are advisory and never touch the ledger.

### Step 6 — Report

```
## /spec-verify auth — 2026-05-14

Code root:  /Users/alice/work/service-api/src/auth/
Test root:  /Users/alice/work/service-api/tests/auth/
Test cmd:   pnpm test auth

Spec @ abc1234   ⇄   Code @ def5678

Conformance:     ✓ 4/4 witness traces replayed green
                   REQ-001 ✓ (4 states)  REQ-002 ✓ (3 states)  REQ-003 ✓ (6 states)  REQ-004 ✓ (4 states)
Completeness:    ✓ 6 PASS, 0 WARN, 0 FAIL
Correctness:     ✓ 4 PASS, 1 WARN (INV-001 guard could be tightened — see below), 0 FAIL
Coherence:       ✓ 3 PASS, 0 WARN, 0 FAIL
Regression:      ✓ 42 passed, 0 failed

Overall:         PASS
Drift detected:  no
```

A conformance failure renders like:

```
Conformance:     ✗ 3/4 green; REQ-003 FAILED at step 5
                   expected accountStatus[bob] = Locked after 5th login_failed
                   observed code state: bob still Unlocked (lockout fires at attempt 6 — '>' vs '>=' in authService.ts:78?)
```

If drift:

```
Overall:         FAIL
Drift detected:  YES

Findings cluster on existing requirements rather than recent additions, and git log
shows commits to traced files (authService.ts) since the last verify.

Two paths:
  1. Code is wrong (regression). Revert the offending commits.
  2. Code is right; spec is stale. Run /spec auth — it'll detect drift and walk
     you through codifying it.
```

### Step 7 — Commit

```bash
git add specs/<target>.*.json
git commit -m "spec(<target>): verify — <pass|fail>, <summary>"
```

Show the commit message; wait for confirmation.

### CI usage

The split (rationale: METHODOLOGY → Git Conventions):

- **Code repo CI**: wire `conformance.command` into the standard test run — the replay harness is just a test file, so a code change that breaks model conformance fails immediately on the code PR, with no spec tooling installed.
- **Spec repo CI** (`spec-ci.yml`): the cheap deterministic gates only — lint, matrix `--strict`, quint typecheck/test.
- **Full `/spec-verify`** (traceability validation, drift detection, verification log) is agent-driven: run it before merge or on a schedule; it is not part of the shipped CI workflow.
